# 🧠 AiEmployee Project — Engineering Summary

## 📅 Date: 2026-04-02

---

# 🎯 Project Overview

**AiEmployee** is a hybrid AI system designed to act as an intelligent agent that can:

* Receive messages (starting with Telegram)
* Analyze conversations between users
* Make judgments (e.g., who is right)
* Return structured reasoning

---

# 🏗️ Architecture

We are using:

* Clean Architecture
* Domain-Driven Design (DDD)
* Channel-agnostic design
* AI as an external tool (NOT core logic)

---

# 🧱 High-Level Structure

```
AiEmployee/
│
├── src/
│   ├── AiEmployee.Api/            → Controllers (Adapters)
│   ├── AiEmployee.Application/    → UseCases + Interfaces (Ports)
│   ├── AiEmployee.Domain/         → Core business models (Pure)
│   └── AiEmployee.Infrastructure/ → External systems (AI, DB)
│
├── ai-service/                    → Python FastAPI (AI layer)
│
├── tests/
│   ├── UnitTests/
│   └── IntegrationTests/
│
└── AiEmployee.sln
```

---

# 🔁 System Flow

```
Client (Postman / Telegram)
    ↓
C# API (Controller)
    ↓
Application Layer (UseCase)
    ↓
AI Port (IAiClient)
    ↓
Infrastructure (HTTP call)
    ↓
Python FastAPI (/ai/judge)
    ↓
AI Response
    ↓
Mapped to Domain Model
    ↓
Returned to client
```

---

# 🧠 Core Design Principles

* Domain must be PURE (no HTTP, no AI, no JSON)
* Application orchestrates logic and calls external services
* Infrastructure handles external integrations (AI, DB)
* AI is replaceable (loose coupling)
* DTOs are used as anti-corruption layer

# Technical Summary — AI Employee Platform (Reliability & Stability Improvements)

---

## 1. Project Context

The project is an AI-powered Telegram-based platform built using Clean Architecture (.NET backend + Python AI service with LLM).

This module focuses on:

* Telegram webhook processing
* AI-based judgment and lead classification
* Reliable communication between .NET and Python AI service

**Main Goal:**
Improve system reliability and correctness by eliminating silent failures, enforcing fail-loud behavior, and introducing resilience for AI communication.

---

## 2. What Was Implemented

### 1. Fail-Loud Error Handling (Webhook Layer)

* Replaced silent exception handling in `TelegramWebhookController`

* Introduced differentiated error handling:

  * `AiServiceException` → user notified + HTTP 200 (prevent retries)
  * Generic exceptions → HTTP 500 (allow retry)

* Added user-facing fallback message via `ITelegramClient`

---

### 2. AI Failure Normalization

* Introduced `AiServiceException` (Application layer)

* Standardized all AI-related failures:

  * HTTP errors
  * Timeouts
  * Invalid JSON
  * Empty responses

* Ensured all recoverable AI failures are propagated consistently to the controller

---

### 3. Removal of Silent / Fake AI Responses

* Eliminated fallback pattern:

  * `return dto ?? new JudgmentResultDto()`

* Replaced with explicit failure:

  * Throw `AiServiceException` on null/invalid responses

* Prevented false-positive successful responses

---

### 4. Retry Mechanism for AI Calls

* Implemented retry logic in `AiClient`:

  * Max 3 attempts
  * Exponential backoff (500ms → 1000ms)
  * Retry only on:

    * `HttpRequestException`
    * `TaskCanceledException`

* No retry for:

  * `JsonException`
  * `AiServiceException`

---

### 5. Error Handling Strategy

* Defined clear separation:

  * **Recoverable errors (AI-related)** → handled gracefully
  * **System errors** → fail loud (HTTP 500)

* Prevented:

  * Retry storms
  * Duplicate message processing
  * Silent data corruption

---

## 3. Technical Details

### Technologies

* Backend: ASP.NET Core (.NET)
* AI Service: Python FastAPI
* LLM Integration: Ollama (local)
* Communication: HTTP (JSON)

---

### Architecture

* Clean Architecture:

  * Domain: pure entities
  * Application: use cases + interfaces
  * Infrastructure: AI client, Telegram client
  * API: controllers

---

### Key Patterns & Decisions

* AI accessed via `IAiClient` abstraction
* Introduced domain-level exception (`AiServiceException`) for boundary control
* Fail-loud design with controlled recovery paths
* Retry implemented manually (lightweight, no Polly)
* No architectural refactor (minimal, targeted changes)

---

### Constraints

* In-memory repositories still in use (.NET side)
* No centralized logging or tracing
* Hardcoded AI base URL (not yet externalized)

---

## 4. Current State

### Fully Working

* Telegram webhook handling with safe error behavior
* AI judgment and lead classification (end-to-end)
* Retry-enabled AI communication
* Consistent error propagation across layers
* User notification on recoverable failures

---

### Stability Improvements

* No silent failures
* No fake/empty AI responses
* No retry storms from Telegram
* Predictable system behavior under failure

---

## 5. Pending Tasks / Next Steps

2. Externalize AI service configuration (base URL, timeouts)
3. Replace in-memory repositories with persistent storage
4. Improve structured logging (add context, correlation IDs)
